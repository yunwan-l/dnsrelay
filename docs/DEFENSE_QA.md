# 老师验收时高频问题准备

## 1. 为什么要做 ID 转换

因为多个客户端可能同时发起查询，而上游 DNS 返回结果时只靠 `ID` 来区分属于哪一个请求。  
如果不做转换和映射，当多个请求并发存在时，就可能把响应发错客户端。

## 2. 为什么用 UDP 还要考虑超时

因为 UDP 不可靠：

- 可能丢包
- 可能乱序
- 可能迟到

如果上游 DNS 长时间不回，程序不能一直等，所以要超时清理并返回 `SERVFAIL`。

## 3. 什么是迟到响应，为什么要丢弃

如果一个中继请求已经超时并从映射表删除，之后上游 DNS 才把结果发回来，这就是迟到响应。  
这时原来的客户端可能早就结束等待了，如果还继续转发，可能会造成逻辑错误，所以要丢弃。

## 4. 为什么使用 select() 而不是循环探测

`select()` 可以在 socket 可读之前休眠等待，不会一直空转占用 CPU。  
这正是老师 PPT 里强调的“不能忙等待”。

## 5. 为什么本工程不再使用两个 socket

老师 PPT 里专门提醒过“两 socket 方案理解不到位”的问题。  
这里改成单 socket：

- 结构更清晰
- 更容易解释
- 更符合课程设计要求

## 6. 拦截域名时为什么返回 NXDOMAIN

因为老师题目里明确要求：如果本地表查到 `0.0.0.0`，则向客户端返回“域名不存在”的报错消息。  
这对应 DNS 响应码 `RCODE = 3`，即 `NXDOMAIN`。

## 7. 为什么本地表命中但不是 A 查询时返回空回答

因为本地表只保存了 IPv4 地址，也就是 A 记录。  
如果客户端查 AAAA、MX 等其他类型，程序不能伪造错误的 A 记录，所以返回无错误但无回答，比构造错误报文更合理。

## 8. 53 端口为什么可能绑定失败

因为：

1. DNS 默认端口就是 53。
2. 该端口可能已被系统服务占用。
3. 在 Windows 上绑定 53 端口通常需要管理员权限。

## 9. 你们的并发是多线程吗

不是。  
这里使用的是基于 `select()` 的事件驱动并发，属于单线程、多请求并发处理模型。

## 10. 你们怎么验证程序是对的

建议回答：

1. `nslookup` 验证功能是否正确。
2. `ping` 或浏览器验证实际名字解析效果。
3. `ipconfig /flushdns` 排除缓存干扰。
4. `Wireshark` 抓包验证 DNS 报文和 ID 转换。
5. 构造超时场景验证异常处理逻辑。
