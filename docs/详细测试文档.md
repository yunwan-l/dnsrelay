# DNS中继服务器详细测试文档

## 1. 测试目标

本文档用于逐项检查DNS中继服务器的功能实现，覆盖课程设计要求中的本地解析、不良网站拦截、DNS中继、多客户端并发、ID转换、UDP超时处理、命令行参数、调试输出，以及扩展实现中的缓存、多上游、统计和日志功能。

建议测试时把命令行窗口截图保存，后续可直接放入课程设计报告。

## 2. 测试环境准备

### 2.1 编译环境

Windows/MinGW：

```powershell
.\build.bat
```

预期结果：

```text
Build succeeded: dnsrelay.exe
```

Linux：

```sh
make linux
```

预期结果：生成 `dnsrelay` 可执行文件。

### 2.2 运行权限

DNS默认监听UDP 53端口。Windows下通常需要用“管理员身份运行”命令行；Linux下通常需要 `sudo`。

如果启动时报错 `Bind port 53 failed`，常见原因：

1. 没有管理员/root权限。
2. 53端口已被其他DNS服务占用。
3. 本机安全软件或防火墙拦截。

### 2.3 推荐启动命令

为了避免默认DNS在不同网络环境中不可达，建议测试时显式指定上游DNS：

```powershell
.\dnsrelay.exe -d 8.8.8.8 dnsrelay.txt
```

如果 `8.8.8.8` 受网络限制，也可以改用：

```powershell
.\dnsrelay.exe -d 114.114.114.114 dnsrelay.txt
```

测试多上游：

```powershell
.\dnsrelay.exe -d 203.0.113.1,114.114.114.114 dnsrelay.txt
```

说明：`203.0.113.1` 是文档保留测试地址，通常不可达，适合测试超时和切换。

### 2.4 测试命令格式

所有 `nslookup` 测试建议显式指定本地DNS服务器：

```bat
nslookup 域名 127.0.0.1
```

这样不需要修改系统DNS设置。

如果测试系统整体DNS解析，则可以把系统DNS设为 `127.0.0.1`，测试结束后记得恢复原DNS。

### 2.5 终端使用规则

Windows PowerShell中运行当前目录的程序时，要在命令前加 `.\`。

正确示例：

```powershell
.\dnsrelay.exe -d 8.8.8.8 dnsrelay.txt
.\build.bat
```

不推荐直接输入：

```powershell
dnsrelay.exe
build.bat
```

原因是 PowerShell 默认不会从当前目录查找可执行文件。

测试时建议固定使用两个终端：

1. 服务器终端：只负责运行 `dnsrelay.exe`，启动后不要关闭。
2. 客户端终端：只负责执行 `nslookup ... 127.0.0.1`。
3. 每次更换服务器启动参数时，先在服务器终端按 `Ctrl+C` 停止旧进程，再启动新的命令。
4. 如果要清空DNS缓存，Windows下最稳的方法是按 `Ctrl+C` 停止服务器后重新启动。

### 2.6 启动命令怎么选

普通功能测试使用一级调试，日志清楚且不太多：

```powershell
.\dnsrelay.exe -d 8.8.8.8 dnsrelay.txt
```

缓存和查询类型隔离测试使用二级调试，因为要看 `CACHE PUT` 和 `CACHE HIT`：

```powershell
.\dnsrelay.exe -dd 8.8.8.8 dnsrelay.txt
```

默认参数测试直接运行程序，不加任何参数：

```powershell
.\dnsrelay.exe
```

单上游超时测试使用一个不可达上游：

```powershell
.\dnsrelay.exe -d 203.0.113.1 dnsrelay.txt
```

多上游切换测试使用“不可达上游 + 可达上游”：

```powershell
.\dnsrelay.exe -d 203.0.113.1,114.114.114.114 dnsrelay.txt
```

程序会把 `192.0.2.0/24`、`198.51.100.0/24`、`203.0.113.0/24` 这三段文档保留地址当作测试黑洞，不真正发出UDP包，用来稳定验证超时和多上游切换。

## 4. 详细测试步骤

### T01 编译测试

命令：

```powershell
.\build.bat
```

预期结果：

```text
Build succeeded: dnsrelay.exe
```

通过标准：

1. 生成 `dnsrelay.exe`。
2. 编译过程没有 error。
3. 最好没有 warning。

如果失败：

1. 确认已经安装 MinGW，并且 `gcc` 在 PATH 中。
2. 确认当前目录是项目根目录。
3. 确认所有 `.c` 和 `.h` 文件都在同一目录。

### T02 帮助参数测试

命令：

```powershell
.\dnsrelay.exe --help
```

预期输出包含：

```text
Usage:
-d
-dd
-l <file>
Default: 202.106.0.20
```

通过标准：

1. 程序打印帮助信息后退出。
2. 不创建socket，不占用53端口。

### T03 默认配置加载测试

目的：确认程序能读取默认表文件，并成功监听53端口。

步骤：

1. 打开服务器终端。
2. 执行：

```powershell
.\dnsrelay.exe -d 8.8.8.8 dnsrelay.txt
```

3. 不要关闭这个终端，观察启动输出。

预期输出包含：

```text
Loaded ... domain records from dnsrelay.txt
DNS Relay Server v2.0
Listening   : port 53
Upstream    : 8.8.8.8:53
Config file : dnsrelay.txt
Debug level : 1
```

通过标准：

1. 成功加载配置文件。
2. 成功监听53端口。
3. 上游DNS显示为命令行指定地址。

### T04 本地普通IP命中测试

目的：确认本地表中配置的普通IP会被直接返回，不转发上游DNS。

前置条件：`dnsrelay.txt` 中有：

```text
11.111.11.111 test1
123.127.134.10 bupt
```

步骤：

1. 服务器终端保持 T03 的命令运行：

```powershell
.\dnsrelay.exe -d 8.8.8.8 dnsrelay.txt
```

2. 新开一个客户端终端。
3. 执行：

```powershell
nslookup test1 127.0.0.1
nslookup bupt 127.0.0.1
```

预期结果：

```text
Name:    test1
Address: 11.111.11.111
```

```text
Name:    bupt
Address: 123.127.134.10
```

程序日志预期包含：

```text
QUERY
HIT    test1 -> 11.111.11.111
```

通过标准：

1. 返回IP与表中配置完全一致。
2. 日志显示 `HIT`。
3. 不应该出现 `RELAY`。

### T05 不良网站拦截测试

目的：确认 `0.0.0.0` 表项不会返回IP，而是返回域名不存在。

前置条件：`dnsrelay.txt` 中有：

```text
0.0.0.0 008.cn
```

步骤：

1. 服务器终端继续保持：

```powershell
.\dnsrelay.exe -d 8.8.8.8 dnsrelay.txt
```

2. 在客户端终端执行：

```powershell
nslookup 008.cn 127.0.0.1
```

预期结果：

```text
*** can't find 008.cn: Non-existent domain
```

或显示类似 `NXDOMAIN`、`不存在的域名`。

程序日志预期包含：

```text
QUERY
BLOCK  008.cn
```

通过标准：

1. 不能返回 `0.0.0.0` 作为A记录。
2. 应返回域名不存在，即 DNS RCODE 为 NXDOMAIN。
3. 日志显示 `BLOCK`。

### T06 DNS中继测试

目的：确认表中没有的域名会转发到外部DNS，并把上游结果返回客户端。

步骤：

1. 如果服务器终端还在运行 `8.8.8.8` 上游，可以继续使用。
2. 如果服务器已停止，在服务器终端启动：

```powershell
.\dnsrelay.exe -d 8.8.8.8 dnsrelay.txt
```

3. 在客户端终端执行：

```powershell
nslookup www.bupt.edu.cn 127.0.0.1
```

预期结果：

```text
Name:    www.bupt.edu.cn
Address: ...
```

程序日志预期包含：

```text
QUERY
NEW TID
RELAY  www.bupt.edu.cn -> 8.8.8.8
RESTORE TID
REPLY  forwarded
```

通过标准：

1. 表中未配置该域名时，程序走 `RELAY`。
2. 能收到上游DNS响应。
3. 客户端能收到最终解析结果。

如果失败：

1. 尝试把上游换成 `114.114.114.114`。
2. 确认本机能访问外网DNS。
3. 确认防火墙没有拦截UDP 53。

### T07 DNS缓存测试

目的：确认第一次查询写入缓存，第二次相同查询直接从缓存返回。

步骤：

1. 在服务器终端按 `Ctrl+C` 停止旧服务器。
2. 使用二级调试重新启动：

```powershell
.\dnsrelay.exe -dd 8.8.8.8 dnsrelay.txt
```

3. 在客户端终端连续执行两次：

```powershell
nslookup www.bupt.edu.cn 127.0.0.1
nslookup www.bupt.edu.cn 127.0.0.1
```

预期日志：

第一次：

```text
RELAY
CACHE PUT
```

第二次：

```text
CACHE
CACHE HIT
```

通过标准：

1. 第一次查询转发上游。
2. 第二次同域名同类型查询直接命中缓存。
3. 客户端两次都能得到响应。

### T08 查询类型隔离测试

目的：确认缓存key包含 QTYPE/QCLASS，不会让A查询和AAAA查询互相污染。

步骤：

1. 服务器终端继续使用二级调试：

```powershell
.\dnsrelay.exe -dd 8.8.8.8 dnsrelay.txt
```

2. 如需清空缓存，先按 `Ctrl+C` 停止服务器，再重新执行上面的启动命令。
3. 在客户端终端执行：

```powershell
nslookup -type=A www.bupt.edu.cn 127.0.0.1
nslookup -type=AAAA www.bupt.edu.cn 127.0.0.1
nslookup -type=A www.bupt.edu.cn 127.0.0.1
```

预期：

1. A查询和AAAA查询分别处理。
2. 第三次A查询可以命中A记录缓存。
3. AAAA查询不会错误返回A记录缓存。

通过标准：

1. 日志中的缓存key类似 `www.bupt.edu.cn|1|1` 和 `www.bupt.edu.cn|28|1`。
2. 不同QTYPE对应不同缓存项。

### T09 多客户端并发测试

目的：确认多个客户端同时查询时，程序不会串包，也不会卡死。

服务器终端：

```powershell
.\dnsrelay.exe -d 8.8.8.8 dnsrelay.txt
```

方法一：推荐使用批处理一键并发测试。

1. 保持服务器终端运行。
2. 新开一个客户端终端。
3. 执行项目中的并发测试脚本：

```powershell
.\test_t09.bat
```

脚本默认会同时打开三个命令行窗口，分别查询：

```text
www.bupt.edu.cn
www.baidu.com
www.qq.com
```

如果需要并发更多域名，打开 `test_t09.bat`，删除备用行开头的 `rem ` 即可启用更多查询。例如：

```bat
rem start "T09-163" cmd.exe /k "nslookup www.163.com 127.0.0.1"
```

改成：

```bat
start "T09-163" cmd.exe /k "nslookup www.163.com 127.0.0.1"
```

方法二：手动测试。

1. 保持服务器终端运行。
2. 打开三个客户端终端。
3. 分别先输入下面三条命令，但先不要按回车。
4. 然后快速依次按回车，让请求尽量重叠。

```powershell
nslookup www.bupt.edu.cn 127.0.0.1
nslookup www.baidu.com 127.0.0.1
nslookup www.qq.com 127.0.0.1
```

说明：不建议使用 PowerShell `Start-Job` 方式截图，因为 `nslookup` 的中文输出可能被 PowerShell 当作错误流显示成红色，容易误判。

预期日志：

```text
QUERY  www.bupt.edu.cn
QUERY  www.baidu.com
QUERY  www.qq.com
NEW TID: orig=... -> new=...
NEW TID: orig=... -> new=...
RESTORE TID: new=... -> orig=...
REPLY  forwarded
```

通过标准：

1. 多个查询都能得到响应。
2. 不出现响应串到其他查询窗口的情况。
3. 程序日志中能看到多个独立的 `NEW TID` 和 `RESTORE TID`。
4. 程序不崩溃，不忙等待。
5. `1.0.0.127.in-addr.arpa` 是 `nslookup` 自动产生的反向解析请求，可以忽略，不影响并发测试结论。

### T10 ID转换测试

目的：确认转发前分配新ID，上游响应回来后恢复客户端原始ID。

步骤：

1. 服务器终端启动一级调试：

```powershell
.\dnsrelay.exe -d 8.8.8.8 dnsrelay.txt
```

2. 客户端终端执行：

```powershell
nslookup -type=A www.bupt.edu.cn 127.0.0.1
```

预期日志：

```text
QUERY  ID=客户端原始ID
NEW TID: orig=客户端原始ID -> new=上游新ID
RESTORE TID: new=上游新ID -> orig=客户端原始ID
```

通过标准：

1. 转发前分配新ID。
2. 收到响应后恢复原始ID。
3. 客户端正常收到响应。

### T11 单上游超时测试

目的：确认只有一个上游DNS且它不可达时，程序不会一直等待，会返回 `SERVFAIL`。

说明：Windows的 `nslookup` 默认2秒超时，而程序默认等待约5秒。因此客户端命令要加 `-timeout=8`，否则客户端可能先超时，服务器还没来得及返回 `SERVFAIL`。

步骤：

1. 在服务器终端按 `Ctrl+C` 停止旧服务器，清空缓存。
2. 启动不可达上游：

```powershell
.\dnsrelay.exe -d 203.0.113.1 dnsrelay.txt
```

3. 在客户端终端执行：

```powershell
nslookup -timeout=8 -type=A www.bupt.edu.cn 127.0.0.1
```

预期：

1. 程序等待约 `TIMEOUT_SEC` 秒。
2. 客户端收到失败响应。
3. 日志出现 `SERVFAIL`。

预期日志：

```text
RELAY  www.bupt.edu.cn -> 203.0.113.1
SERVFAIL www.bupt.edu.cn after 1 attempt(s)
```

通过标准：

1. 程序不会一直挂起请求。
2. 不会崩溃。
3. 统计中的 `Relay timeouts` 增加。

如果日志里仍然出现 `REPLY forwarded`，先确认已经重新编译并重启了最新的 `dnsrelay.exe`。旧版本会真实发包，可能被当前网络透明代理回答。

如果只有 `1.0.0.127.in-addr.arpa` 出现 `SERVFAIL`，而目标域名仍然出现 `REPLY forwarded`，说明反向解析请求超时了，但目标域名没有超时。此时不能算 T11 通过，必须看到目标域名本身出现 `SERVFAIL`。

### T12 多上游切换测试

目的：确认第一个上游不可达时，程序切换到第二个上游并最终返回结果。

步骤：

1. 在服务器终端按 `Ctrl+C` 停止旧服务器，清空缓存。
2. 启动两个上游，第一个不可达，第二个可达：

```powershell
.\dnsrelay.exe -d 203.0.113.1,114.114.114.114 dnsrelay.txt
```

3. 在客户端终端执行：

```powershell
nslookup -timeout=8 -type=A www.bupt.edu.cn 127.0.0.1
```

预期日志：

```text
RELAY  www.bupt.edu.cn -> 203.0.113.1
UPSTREAM Failover: 203.0.113.1 -> 114.114.114.114
TIMEOUT www.bupt.edu.cn, retry upstream #1
REPLY  forwarded
```

通过标准：

1. 第一个上游不可达时自动切换。
2. 第二个上游可达时最终解析成功。
3. 客户端能收到响应。

严格通过标准：`RELAY`、`Failover`、`TIMEOUT`、最终 `REPLY forwarded` 最好对应同一个测试域名。如果只在 `1.0.0.127.in-addr.arpa` 这种反向解析请求上出现切换，说明功能被触发过，但不适合作为主测试截图。

如果第一个上游看起来不可达却仍然直接返回结果，先确认已经重新编译并重启了最新的 `dnsrelay.exe`。最新版本会把 TEST-NET 地址作为测试黑洞，不会真实发包。

### T14 日志文件测试

启动：

```powershell
.\dnsrelay.exe -d -l run.log 8.8.8.8 dnsrelay.txt
```

执行：

```bat
nslookup test1 127.0.0.1
nslookup 008.cn 127.0.0.1
```

检查：

```bat
type run.log
```

预期：

1. `run.log` 被创建。
2. 文件中包含启动时间、QUERY、HIT、BLOCK等日志。

通过标准：

1. 控制台和日志文件都有主要输出。
2. 日志文件可用于报告截图或结果记录。

### T15 配置格式兼容测试

新建 `format-test.txt`：

```text
1.1.1.1 ipfirst
domainfirst 2.2.2.2
0.0.0.0 block-ipfirst
block-domainfirst 0.0.0.0
example.com 3.3.3.3
```

说明：这个测试文件同时包含本地普通 A 记录和拦截记录，`example.com` 用来演示修改配置后无需重启即可生效。

启动：

```powershell
.\dnsrelay.exe -d 8.8.8.8 format-test.txt
```

测试 1：本地 A 记录命中

```powershell
nslookup -type=A ipfirst 127.0.0.1
```

预期：返回 `1.1.1.1`，日志显示 `HIT    ipfirst -> 1.1.1.1`。

测试 2：AAAA 查询转发上游

```powershell
nslookup -type=AAAA ipfirst 127.0.0.1
```

预期：日志显示 `RELAY  ipfirst -> 8.8.8.8`，返回上游 AAAA 结果或无答案，但不会把 `1.1.1.1` 当成 AAAA 返回。

测试 3：MX 查询转发上游

```powershell
nslookup -type=MX ipfirst 127.0.0.1
```

预期：日志显示 `RELAY  ipfirst -> 8.8.8.8`，返回上游 MX 结果或无答案，但不会把 `1.1.1.1` 当成 MX 返回。

测试 4：拦截记录生效

```powershell
nslookup -type=A block-ipfirst 127.0.0.1
nslookup -type=AAAA block-ipfirst 127.0.0.1
nslookup -type=MX block-ipfirst 127.0.0.1
```

预期：都返回 NXDOMAIN，日志显示 `BLOCK  block-ipfirst`。

测试 5：配置修改后自动生效

1. 将 `example.com 3.3.3.3` 改成 `0.0.0.0 example.com` 并保存。
2. 不重启程序，直接执行：

```powershell
nslookup -type=A example.com 127.0.0.1
nslookup -type=AAAA example.com 127.0.0.1
nslookup -type=MX example.com 127.0.0.1
```

预期：A/AAAA/MX 都按最新配置处理；日志里应先出现重载提示，再出现新的 `BLOCK` 或 `RELAY` 结果。

通过标准：

1. 两种格式都能加载。
2. A、AAAA、MX 三种查询能按 QTYPE 分别验证。
3. 拦截记录在三种类型下都能生效。
4. 修改配置文件后无需重启即可生效。

## 6. 常见问题排查

### 6.1 无法绑定53端口

现象：

```text
Bind port 53 failed
```

处理：

1. Windows用管理员身份运行命令行。
2. Linux使用 `sudo`。
3. 检查是否已有DNS程序占用53端口。

### 6.2 中继查询失败

处理：

1. 换上游DNS：`8.8.8.8`、`114.114.114.114`、本机 `ipconfig /all` 查到的DNS。
2. 检查本机网络是否允许UDP 53。
3. 暂时关闭防火墙或安全软件测试。

### 6.3 `nslookup` 结果使用了别的DNS

确保命令最后带 `127.0.0.1`：

```bat
nslookup test1 127.0.0.1
```

如果改了系统DNS，测试结束后恢复原DNS。

### 6.4 缓存测试看不到 `CACHE`

处理：

1. 使用 `-dd` 启动，查看详细日志。
2. 确保两次查询域名、类型一致。
3. 第一次查询必须是上游成功响应，才会写入缓存。
4. TTL过短或响应无答案区时可能不会缓存。

### 6.5 多上游切换不成功

处理：

1. 确认命令中有多个上游，逗号中间不要有中文逗号。
2. 示例：`203.0.113.1,114.114.114.114`
3. 确认第二个上游本身可达。
